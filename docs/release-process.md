# 发布规范

本文档定义 WeSight 的最小发布流程。

## 发布门禁

发布稳定版前，必须运行并记录结果：

```bash
npm run build
npm run lint
npm test
```

任一门禁失败时，不得发布稳定版。若仍需要共享候选版本，必须把失败项写入已知问题或阻断项。

## Changelog

每个版本都必须在 `CHANGELOG.md` 中记录：

- 版本
- 日期
- 新增
- 变更
- 修复
- 已知问题

GitHub release note 应从 `CHANGELOG.md` 的对应版本条目生成。

## 发布审查

打 tag 前确认：

- 版本号与包元数据一致。
- 已知问题明确、可行动。
- 发布说明只描述对开发者或用户有意义的项目级变化。
- 涉及安全、签名、校验和平台差异时，已有对应检查记录。
